# Avium (아비움) Phase 1 PRD v1.2

**Offline Food Safety Lookup for Parrot Caregivers**

## 0. 문서 메타

- 문서명: Avium Phase 1 PRD (Offline Food Safety Lookup)
- 버전: v1.2
- 작성/개정일: 2026-02-14
- 제품 목표: **오프라인에서 “이 음식 급여 가능?”을 3초 내 판단** + **오해(검색 0건, 조건 미선택, 의료행위 기대)로 인한 사고 최소화**
- Phase 1 범위 고정 원칙:
  - 서버/로그인/동기화 없음
  - 오프라인 DB 번들 포함
  - 진단/치료가 아닌 참고 정보 + 우선순위 안내

## 1. 배경 및 문제 정의

### 1.1 사용자 문제

앵무새 집사들은 다음 질문을 반복한다.

- “이 음식 먹여도 되나요?”
- “조리/가공/부위에 따라 달라지나요?”
- “실수로 먹었는데 지금 뭐 해야 하나요?”

실제 사고 패턴(제품 리스크 최상위):

1. 앱에 없음(검색 0건)
2. “그럼 괜찮나?”로 오해
3. 급여
4. 문제 발생

따라서 **검색 0건은 ‘안전 아님’을 강제적으로 전달**해야 한다.

### 1.2 제품이 제공하는 가치

- **오프라인에서도 즉시 판단** 가능한 “안전도 배지 + 한 줄 요약”
- **조건부 안전도(부위×조리)**를 구조화하여 “텍스트에 숨은 조건”을 제거
- 실수 섭취 시, 치료 지시가 아니라 **행동 우선순위 + 관찰 체크리스트 + 진료 권고 트리거** 제공

## 2. 목표 및 핵심 원칙

### 2.1 Phase 1 제품 목표

- 음식 안전 여부를 **3초 내 확인**
- **오프라인 완전 동작**
- 입력 부담 최소화(조건은 “필요할 때만” 선택)
- 의료행위(진단/치료/정량 처치)로 오해받지 않는 표현과 UX
- 조건부 안전도를 **구조적으로, 오해 없이** 표시
- 검색 0건에서 “안전 오해”를 **UX로 차단**

### 2.2 Phase 1 핵심 원칙

- **First View에서 판단**: 결과 리스트/상세 상단에서 결론이 보인다.
- **보수적 표현**: 애매하면 위험을 과소평가하지 않는다.
- **조건 선택 미완료는 ‘확정’처럼 보이지 않게**: 부분 선택은 경고 포함.
- **검색 0건 = 안전 아님(P0)**: 상단 고정 문구 + 2개 액션만 제공.
- **서버 없음**: DB는 번들, 업데이트는 앱 업데이트로만 배포.

## 3. 대상 사용자/페르소나 및 핵심 시나리오

### 3.1 페르소나(Phase 1)

- 초보 집사: 검색 결과의 “배지/한 줄”만 보고 판단하는 비율이 높음
- 중급 집사: 조건(부위/조리/가공)을 확인하고 싶어함
- 응급 상황 사용자: “치료법”이 아니라 **지금 당장 해야 할 우선순위**를 원함

### 3.2 핵심 시나리오

1. 음식 이름 검색 → 리스트에서 배지 보고 즉시 판단
2. 상세 진입 → 조건(부위×조리) 선택 → 조건별 안전도 확인
3. 실수 섭취 → 응급 모드 → 위험도(권장 대응 긴급도) + 관찰 체크 + 진료 권고
4. 검색 0건 → “안전 아님” 고지 → (음식 미상) 응급 모드 또는 정보요청 메일

## 4. 등급 체계 및 용어 표준

### 4.1 안전도 등급(콘텐츠 등급)

- **Safe**: 일반적인 급여 범위에서 문제 보고가 적은 편이며, 안내된 팁을 따르면 위험이 낮은 편
- **Caution**: 개체/부위/가공/섭취량·빈도 등에 따라 문제가 발생할 수 있어 **조건 안내가 필수**
- **Danger**: 조류에서 독성 또는 중증 위험이 보고되었거나 소량에서도 문제가 될 가능성이 있어 **급여하지 않는 것을 기본**

### 4.2 표시 라벨(UX) vs 데이터 enum(스키마) 표준(P0)

- **표시 라벨(UX 텍스트)**
  - 안전도: `Safe / Caution / Danger`
  - 응급 위험도: `Low / Medium / High`

- **데이터 enum(로직 비교용)**
  - 안전도: `"safe" | "caution" | "danger"`
  - 응급 위험도: `"low" | "medium" | "high"`

- 목적: 대소문자 혼선으로 인한 버그/정렬 오류 방지

### 4.3 “대표 배지(safetyLevel)” 규칙

- `safetyConditions`가 비어 있으면: **`safetyLevel` 그대로 대표 배지**
- `safetyConditions`가 존재하면:
  - 모든 조건 레벨이 동일하면: 그 레벨을 대표 배지로 사용 가능
  - 하나라도 다르면: 대표 배지는 **기본 `caution`** (조건 확인 필요)

## 5. 기능 요구사항

## 5.1 음식 검색

### 5.1.1 동작 개요

- 입력: 사용자가 검색창에 음식명을 입력(한글/영문/별칭/띄어쓰기 변형/오타)
- 출력: 결과 리스트에서 **배지 + 한 줄 요약**으로 즉시 판단

### 5.1.2 검색 요구사항

- 한글/영문 검색 지원
- `aliases` 포함 검색
- 자동완성 제공
- 퍼지 검색 허용(오타/띄어쓰기/복합어)
- 오프라인 로컬 JSON 기반
- 성능: **검색 응답 300ms 이내(기준 디바이스에서)**

### 5.1.3 정규화(normalization)

- 소문자화
- 공백 제거/축약
- 특수문자 제거(최소한의 범위)
- 한/영 alias 포함

### 5.1.4 정렬 우선순위

1. prefix 매칭
2. contains 매칭
3. fuzzy fallback(후보 상위 N개 제한 후 스코어링, 예: N=30)

### 5.1.5 derived field 정책(P0)

- 편집자가 작성하는 필드: `nameKo`, `nameEn`, `aliases`
- `search.tokens`는 **빌드 타임 자동 생성되는 derived field**
- 편집 프로세스에서 `search.tokens`를 수동 편집하지 않는다.

### 5.1.6 검색 0건 UX (P0, 필수)

검색 결과가 0건이면, 화면 구조는 아래로 고정한다.

**(1) 상단 고정 1줄(퍼스트 뷰, 가장 눈에 띄게)**

- **“DB에 없는 음식은 안전하다는 뜻이 아닙니다.”**

**(2) 보조 고지(2~3줄 이내)**

- “성분/가공/첨가물/급여량을 확신하기 어렵다면 **caution으로 간주**하세요.”
- “불안하거나 증상이 있으면 **진료기관 문의를 권장**합니다.”

**(3) 버튼은 2개만(Primary action 혼선 방지)**

1. **응급 모드 열기(음식 미상)**
2. **정보 요청 템플릿(메일)**
   - 서버 전송 없음
   - 사용자의 메일 앱에 템플릿만 생성

**(4) 선택 요소(버튼 아래, Secondary)**

- “비슷한 검색어” 제안(탭 시 검색어만 채움)

## 5.2 음식 상세 화면

### 5.2.1 First View(상단 영역) 구성(필수)

- 음식명(한글/영문)
- 대표 배지(🟢/🟡/🔴 + 라벨)
- 한 줄 요약
- **검토일 + 참고 정보 최소 고지(스크린샷 대응, P1 부분수용)**
  - 예: `검토: 2026-02-01 · 참고 정보 ⓘ`
  - ⓘ 탭: “본 내용은 참고 정보이며 진단/치료를 대체하지 않습니다.”

### 5.2.2 조건 선택 UX (부위 × 조리 2차원 고정, P0/P1)

조건이 존재하는 음식은 조건칩을 **그룹 분리**해서 노출한다.

- 그룹 A: **부위(part)**
- 그룹 B: **형태·조리(prep)**

#### 조건 선택 상태 정의

- **완전 선택(Complete)**: part와 prep가 모두 선택됨
  → 조건 매칭을 수행하고, 조건 레벨을 “조건 반영 결과”로 표시 가능
- **미완전 선택(Incomplete)**: part 또는 prep 중 하나만 선택됨
  → 오해 방지 규칙 적용(아래)

#### 미완전 선택 오해 방지 규칙(P0)

- 대표 배지(`safetyLevel`)는 유지(대개 `caution`)
- 배지 근처에 짧은 경고 텍스트:
  - “조건 선택이 완전하지 않아 보수적으로 표시됩니다.”

- 조건 노트는 “선택된 조건 기준”임을 명시:
  - 예: “현재 선택: 과육 / 조리 미선택”

#### 애매 매칭 보수성(P0)

- 조건 매칭이 복수 후보/애매함이면:
  - 결과 레벨은 **항상 보수적으로 상향**: `danger > caution > safe`
  - UI는 “확정”처럼 보이지 않게(미완전 선택 경고 유지)

### 5.2.3 상세 섹션 구성

1. **왜? (근거 요약)**: 2~3줄, 단정 금지
2. **조건(특히 Caution 필수)**: 부위/가공/양/빈도 템플릿
3. **급여 팁**: allowedParts / avoidParts / frequency / notes / examplesKo
4. **혼동 주의(confusables)**: 이름이 비슷한 재료/품종 혼동 방지
5. **근거/검토 정보(접기/펼치기)**: evidenceLevel, reviewedAt, sources 요약
6. **응급 버튼**: “실수로 먹었어요” → 해당 음식 선택 상태로 응급 모드 진입

### 5.2.4 혼합/가공 고정 경고 컴포넌트(P1)

- `foodType`가 `"mixed"` 또는 `"processed"`이면 상세 상단에 **고정 경고 블록** 노출(반복 템플릿 사용)
- 사용자가 한 줄 결론만 보고 이탈해도 위험 경고가 남도록 설계

## 5.3 응급 모드 (Emergency Mode)

### 5.3.1 기능 목적

- 진단/치료가 아니라 **권장 대응의 긴급도 안내**
- **관찰 체크리스트 + 진료 권고 트리거** 제공

### 5.3.2 정책(필수)

- 금지: 투약/해독/구토 유도/가정 처치 지시/치료법 제시
- 필수: 상단 고정 문구
  - “조류 진료 가능 병원/수의사 연락을 권장합니다.”

### 5.3.3 입력

- 먹은 음식 선택
  - 상세에서 진입 시 자동 선택
  - 검색 0건에서 진입 시 **음식 미상(unknownFood)** 지원

- 섭취 시간(선택): 방금 / 1시간 이내 / 6시간 이내 / 모름
- 증상 체크(선택)
- 조리/형태 선택(선택, 해당 음식에 조건이 있을 때만)

> 체중 입력: Phase 1 제외(의료 오인 및 정량 기대치 상승 리스크)

### 5.3.4 출력

- `riskLevel`: Low/Medium/High (표시 라벨), 내부 enum은 `"low"|"medium"|"high"`
- 즉시 행동 가이드(우선순위 1~3개)
- 관찰 증상 목록
- 진료 권장 문구(트리거 포함)
- 금지 고지(짧게 재노출)

### 5.3.5 룰 기반 산정(테스트 가능 형태, P0)

- 엔진: `ruleBased`
- 입력(내부 정규화):
  - `baseRisk` (food.emergency.baseRisk, 음식 미상은 기본 `"medium"`)
  - `timeBucket`: `"justNow" | "within1h" | "within6h" | "unknown"`
  - `hasTriggerSymptom`: boolean
  - `symptomCount`: number

#### 트리거 증상(Phase 1 고정)

- 호흡 이상
- 경련
- 의식 저하/반응 없음
- 급격한 무기력(움직이지 않음 수준)
- 지속 구토/역류(반복)

#### 룰(우선순위 상단부터 적용)

1. `hasTriggerSymptom = true` → `riskLevel = "high"`
2. else if `baseRisk = "high"` → `riskLevel = "high"`
3. else if `baseRisk = "medium"` and `symptomCount >= 2` → `riskLevel = "high"`
4. else if `baseRisk = "medium"` and `symptomCount = 1` → `riskLevel = "medium"`
5. else if `baseRisk = "low"` and `symptomCount >= 2` → `riskLevel = "medium"`
6. else → `riskLevel = baseRisk`(단, baseRisk가 `"medium"`이면 medium 유지)

#### 시간 모름 보수 문구 규칙

- `timeBucket = "unknown"`이면:
  - riskLevel을 강제로 올리진 않더라도
  - **진료 권고 문구 강도를 한 단계 보수적으로 출력**

## 5.4 기본 급여 가이드

### 5.4.1 Phase 1 종 그룹

- 소형 앵무 / 중형 앵무 / 대형 앵무

### 5.4.2 제공 정보(요약)

- 펠렛 60~80%
- 신선식 15~30%
- 씨앗 5~10%
- 초보자 5가지 기본 원칙

### 5.4.3 오해 방지 문구

- “본 가이드는 입문자를 위한 일반 원칙이며, 종/연령/질환/개체차에 따라 달라질 수 있습니다.”
- 로리/로리킷 등 식성이 뚜렷한 그룹은 Phase 2 확장 대상으로 정의

### 5.4.4 비정량 예시 정책(P1 부분수용)

- Phase 1에서 **정량(그램/체중 기반/용량 계산)**은 제공하지 않는다.
- 대신 “과잉급여 오해”를 줄이는 **행동 가이드 예시**를 허용한다.
  - “간식으로만(주식 대체 X)”
  - “처음 급여는 한 입 이하로 시작”
  - “이상 반응 시 중단 및 문의 권장”

## 6.2 FoodItem 스키마

### 6.2.1 enums

- `safetyLevel`: `"safe" | "caution" | "danger"`
- `evidenceLevel`: `"high" | "medium" | "low"`
- `foodType`: `"whole" | "mixed" | "processed"`
- `category`: Phase 1에서는 문자열(예: `"fruit"`, `"vegetable"`, `"grain"`, `"nut"`, `"drink"`, `"other"`)로 운용하되, 추후 enum 고정 가능

  "oneLinerKo": "과육은 대체로 안전하지만 씨앗은 제거가 권장됩니다.",

## 6.3 SafetyCondition 스키마 + 매칭 규칙(P0)

### 6.3.1 enums

- `part`: `"flesh" | "seeds" | "peel" | "pit" | "leaf" | "stem" | "sprout" | "unknown"`
- `prep`: `"raw" | "cooked" | "dried" | "juice" | "any"`
- `level`: `"safe" | "caution" | "danger"`

```json
{
  "labelKo": "껍질(생)",
  "part": "peel",
  "prep": "raw",
  "level": "caution",
  "noteKo": "껍질은 잔류물/가공 상태에 따라 주의가 필요할 수 있습니다."
}
```

### 6.3.2 조건 매칭 알고리즘(클라이언트 공통, 필수)

**입력**

- `selectedPart` (string|null)
- `selectedPrep` (string|null)

**완전 선택일 때(둘 다 선택됨) 우선순위**

1. `part == selectedPart` AND `prep == selectedPrep`
2. `part == selectedPart` AND `prep == "any"`
3. `part == "unknown"` AND `prep == selectedPrep`
4. `part == "unknown"` AND `prep == "any"`
5. 매칭 실패 → 대표 배지 유지 + “조건 미확정” 표기

**복수 매칭 시**

- 레벨을 **보수적으로 선택**: `danger > caution > safe`

**미완전 선택일 때(하나라도 null)**

- 대표 배지 유지
- “조건 선택 미완료” 경고 표시
- 선택된 조건 기준 텍스트는 가능하되 “확정”처럼 보이지 않게

## 7. 혼합/가공 음식 처리(사고 빈발 구간, P1)

### 7.1 분류 정책

- 성분이 불명확하거나 첨가물 가능성이 큰 항목은 기본 `caution`
- 조류에서 위험 성분이 흔히 포함되는 카테고리는 문구를 더 보수적으로

### 7.2 상세 화면 고정 경고 컴포넌트(필수 템플릿)

`foodType = "mixed" | "processed"`인 항목은 아래를 상세 상단에 고정 표시한다.

- 제목: “혼합/가공식품 주의”
- 문구(불릿):
  - “혼합/가공식품은 성분·첨가물·염분·지방이 변동될 수 있습니다.”
  - “문제가 되는 성분군(카페인/초콜릿/알코올/자일리톨 가능 제품군/양파·파/마늘/고염/고지방/매운 양념/유제품 등)이 섞이기 쉽습니다.”
  - “조합이 불명확하면 **급여하지 않는 것을 기본**으로 하세요.”
  - “확신이 없거나 증상이 있으면 **진료기관 문의**를 권장합니다(응급 모드 연결).”

## 8. 콘텐츠 거버넌스 및 편집 정책

### 8.1 톤/표현 가이드

- 단정 금지: “~로 알려져 있습니다 / ~가 보고됩니다 / ~가능성이 있습니다”
- 과소평가 금지 + 공포 조장 지양
- 조건부 안전도는 가능한 경우 `safetyConditions`로 구조화

### 8.2 evidenceLevel 내부 기준(편집자용)

- `high`: 교과서/기관 가이드/다수 출처 합치
- `medium`: 신뢰 출처 있으나 조건/개체차 여지 큼
- `low`: 제한된 보고/출처 부족(사용자 안내는 더 보수적으로)

### 8.3 편집 프로세스

- 작성 → 내부 리뷰 → 릴리즈(앱 업데이트)
- DB 변경은 `dataVersion/reviewedAt`에 반영
- `search.tokens`는 편집물 아님(derived) — 빌드 파이프라인에서 생성/검증

## 9. 비기능 요구사항(NFR)

### 9.1 오프라인/배포

- 서버 동기화 없음
- DB는 앱 번들 포함
- 업데이트는 앱 업데이트로만 반영
- 설정 화면에서 `dataVersion/reviewedAt` 확인 가능

### 9.2 성능

- 최초 로딩 2초 이내(기준 디바이스)
- 검색 응답 300ms 이내(기준 디바이스)
- 퍼지 검색 후보 N 제한(예: 30)

### 9.3 접근성/디자인

- 색상만으로 상태를 구분하지 않기(아이콘+텍스트 병행)
- 시스템 폰트 스케일 대응
- 다크모드 지원

### 9.4 개인정보

- 로그인 없음
- 개인정보 수집 없음
- 메일 템플릿은 사용자의 메일 앱 호출만(서버 전송/저장 없음)

## 10. 성공 기준 및 QA 정의

### 10.1 사용자 플로우 기준

- 홈 → 검색 → 결과 리스트에서 배지 확인(2탭 내)
- 홈 → 검색 → 상세 진입 후 배지/조건 확인(3탭 내)

### 10.2 “3초 내 판단” 측정 정의

- 기준 디바이스에서:
  - 입력 완료 후 결과 리스트 첫 화면 배지 표시까지 체감 지연이 3초를 넘지 않도록

### 10.3 검색 품질 지표(내부 QA)

- 대표 200개 음식에 대해 쿼리 세트 구성:
  - 정확 키워드 / 띄어쓰기 변형 / 오타 1~2글자 / 영문 / 별칭

- **Top1 노출률**, **Top3 노출률** 측정 및 릴리즈 기준선 설정
- 검색 0건 쿼리:
  - “DB에 없는 음식은 안전 아님” 고지가 **퍼스트 뷰에 노출**되는지 스크린샷 체크리스트로 검수

### 10.4 응급 모드 QA(리그레션 가능)

- 트리거 증상 체크 시 무조건 High로 나오는지
- baseRisk/timeBucket 조합별로 riskLevel 및 문구 강도가 일관되는지
- 금지 문구(가정 처치/투약 금지)가 항상 보이는지

- 최초 실행: “참고 정보이며 진단/치료를 대체하지 않음”

# 13. 이번 리뷰 수용/부분수용/불수용 결정표 (요청한 “이유/근거” 포함)

## 13.1 수용(P0/P1)

- **검색 0건 = 안전 아님** 고지 + **2버튼 UX**
  - 근거: 실제 사고 패턴을 UX로 차단해야 함(가장 큰 제품 리스크)

- **조건부 안전도 2차원(부위×조리) UX 규칙**
  - 근거: “부분 조건 반영”이 오해를 유발 → 미완전 선택 경고 + 보수적 처리 필요

- **혼합/가공 상세 상단 고정 템플릿 컴포넌트**
  - 근거: 실제 사용자는 한 줄 결론만 보고 이탈 → 반복 경고가 사고 감소에 유효

- **라벨(UX) vs enum(데이터) 표준화**
  - 근거: 문자열 비교/정렬/표시 혼선으로 인한 버그 방지(개발 안정성 P0)

- **safetyConditions 매칭 규칙을 PRD에 명시**
  - 근거: 클라이언트별 구현 편차가 생기면 동일 DB가 다른 결과를 내는 위험

- **search.tokens derived field(빌드 생성)로 고정**
  - 근거: 수동 관리 시 누락/중복/품질 편차로 유지보수 붕괴

- **confusables 스키마 추가**
  - 근거: UI 섹션이 있으면 데이터 형식도 고정해야 일관 렌더링 가능

- **응급 모드 룰 기반을 테스트 가능 형태로 명세**
  - 근거: QA/리그레션 가능해야 권고 톤/긴급도 일관성 유지

- **DB 업데이트 정책(서버 없음, 앱 업데이트로 배포) 명시**
  - 근거: 신뢰/CS/법적 리스크 대응 및 기대치 관리

## 13.2 부분수용(제한 포함)

- **급여 팁의 “비정량 예시”**
  - 수용 범위: “간식으로만”, “처음 한 입 이하” 같은 **행동 가이드**
  - 제외 범위: 체중 기반/그램/정량 계산/용량 가이드(의료 오인 및 책임 리스크)

- **스크린샷 유통 대응 고지**
  - 수용 범위: 상세 상단에 `reviewedAt + 참고 정보 ⓘ` **최소 형태** 상시 노출
  - 제한 이유: First View 과밀 방지(3초 판단 원칙 유지)

## 13.3 불수용(없음)

- 이번 라운드에서는 “완전 불수용” 항목은 없음. 다만 위 2개는 Phase 1 리스크 때문에 **명확히 제한**함.
